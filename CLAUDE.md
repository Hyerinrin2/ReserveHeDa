# ReserveHeDa
- 스터디룸 예약 웹 서비스

## 기술 스택
- Java 21
- Spring Boot 3.5.10
- Spring Data JPA
- Spring Security
- PostgreSQL
- Lombok
- Gradle
- Docker + AWS (배포)
- HTML/CSS/JS (프론트)

## 패키지 구조
```
com.heda.BookMark
├── User
│   ├── UserEntity
│   ├── UserDto
│   ├── UserRepository
│   ├── UserService (예정)
│   ├── UserController (예정)
│   ├── LoginType (Enum: LOCAL, KAKAO, NAVER, GOOGLE)
│   └── Role (Enum: ADMIN, USER)
├── Reservation (예정)
│   ├── ReservationEntity
│   ├── ReservationDto
│   ├── ReservationRepository
│   ├── ReservationService
│   ├── ReservationController
│   └── SeatPass (Enum: THREE_HOURS, FIVE_HOURS, SEVEN_HOURS, ALL_DAY)
└── StudyRoom (예정)
    ├── StudyRoomEntity
    ├── StudyRoomDto
    ├── StudyRoomRepository
    ├── StudyRoomService
    └── StudyRoomController
```

## ERD
```
User (1) ──── (N) Reservation (N) ──── (1) StudyRoom
```
- User → Reservation: 한 유저가 여러 예약 가능
- StudyRoom → Reservation: 한 좌석에 여러 예약 가능 (날짜/시간 다르면)
- Reservation.startTime + SeatPass로 endTime 자동 계산

## 코딩 컨벤션
### 브랜치 전략
- main: 최종 배포 (PR로만 머지, 직접 push 금지)
- develop: 개발 통합 (PR로만 머지, 직접 push 금지)
- feature/*: 기능 개발 (작업할 때마다 생성, 머지 후 삭제)
- PR 흐름: feature/* → develop → main

### 브랜치 보호 규칙(필수로 지킬 것)
#### main 브랜치에 직접 작업 절대 금지
- git push origin main 금지
- pr 없이 main에 바로 merge 금지
- pr은 깃헙 웹에서 확인 후 merge 할 것

#### 브랜치 커밋 순서
- feature/기능명 → develop → main
- 기능 구현 후 develop으로 커밋 후 웹에서 merge
- main은 develop에서 최종 작업 후 마지막에 커밋

### pr 우선순위
- P1 (Critical/심각): 즉각적인 조치 필요함(로직 오류 같은 치명적인 오류)
- P2 (High/높음): 반드시 반영 필요(에러나 버그 발생 가능성)
- P3 (Medium/보통): 주요 개선사항
- P4 (Low/낮음): 경미한 영향, 사소한 불편
- P5 (Trivial/미미): 제안사항

### 커밋 메시지
- feat: 기능 구현
- fix: 버그 수정
- refactor: 리팩토링
- test: 테스트 코드
- docs: 문서
- chore: 설정 변경
※ feat랑 test는 커밋을 쪼개서 따로 작성
